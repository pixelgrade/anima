## Part B: Validation Checks

### Check 1: Editor loads without JS errors

**Steps:**
1. Open Chrome DevTools Console tab, filter to Errors
2. Navigate to `http://style-manager.local/wp-admin/post-new.php`
3. Wait for full load, type a few words in title and paragraph
4. Check the console

**Also test:** existing post editor, Site Editor (`site-editor.php`)

**Pass:** Zero JS errors. Warnings acceptable only if pre-existing WP core warnings.

**Failure indicators:**
- `TypeError: Cannot read properties of undefined (reading 'styles')` → Phase 1.1 broke
- `wp-editor is not a registered script` → Phase 1.2 issue
- `SecurityError: Blocked a frame with origin` → Phase 1.5 incomplete

---

### Check 2: SM dynamic CSS variables render in editor canvas

**Steps:**
1. Open a post in the editor
2. In DevTools Elements panel, find `<iframe name="editor-canvas">`
3. Switch to the iframe's document context
4. Inspect `<head>` for SM style tags

**Quick console check (in iframe context):**
```js
getComputedStyle(document.querySelector('.editor-styles-wrapper')).getPropertyValue('--sm-current-bg-color')
// Should return a color value, not empty string
```

**Pass:** SM `<style>` tags present in iframe `<head>` with `--sm-*` custom properties resolving to actual values.

**Failure indicators:**
- No SM styles in iframe `<head>` → Phase 1.1 not reaching iframe
- SM styles in outer `<head>` but not iframe → wrong enqueue context
- Properties resolve to empty → `get_dynamic_style()` output is empty

---

### Check 3: SM color/font controls in Customizer work with live preview

**Steps:**
1. Open `http://style-manager.local/wp-admin/customize.php`
2. Open Style Manager > Colors, change the palette
3. Observe preview pane (should update within ~200ms, no full reload)
4. Open Fonts, change the font palette
5. Observe preview pane (font family should visibly change)
6. Check console throughout

**Pass:** Immediate live preview for both colors and fonts. No JS errors. After "Publish", frontend matches.

**Failure indicators:**
- Preview goes blank → cross-frame error (Phase 1.5)
- `Cannot read properties of undefined (reading 'config')` → `parent.styleManager` access failed
- Colors update but fonts don't → partial fix
- `getElementByTag is not a function` → Phase 1.6 fix not applied

---

### Check 4: Dark mode toggle applies to outer document and editor iframe

**Prerequisite:** Dark mode = "on" in Customizer.

**Steps:**
1. Open a post in the editor
2. In DevTools, check outer `<html>` for `is-dark` class
3. Navigate into `<iframe name="editor-canvas">`, check inner `<html>` for `is-dark` class

**Pass:** `is-dark` present on BOTH outer and inner `<html>` elements. Editor canvas has dark background/light text.

**Failure indicators:**
- `is-dark` on outer but not iframe → Phase 1.4 MutationObserver not working
- `is-dark` appears then disappears from iframe → iframe being recreated, observer not re-applying
- Neither has `is-dark` → Phase 1 inadvertently broke existing dark mode logic

**Dynamic test (dark mode = "auto"):**
1. DevTools > Rendering > Emulate `prefers-color-scheme: dark`
2. Both documents should gain `is-dark`
3. Switch to `light` → both should lose it

---

### Check 5: No Font Library duplicate UI

**Steps:**
1. Open Site Editor (`site-editor.php`)
2. Look for any "Fonts" or "Typography" management panel (WP Font Library feature)
3. In post editor, select a Paragraph block, check Typography section for font options

**Pass:** No standalone Font Library panel visible when SM is active. Font choices come from SM palette only.

**Failure indicators:**
- "Manage Fonts" panel appears alongside SM → Phase 1.7 deregistration didn't work
- PHP error mentioning `wp_unregister_font_collection` → function doesn't exist (needs `function_exists` guard)

---

### Check 6: No cross-frame SecurityError on any admin page

**Steps — visit each with Console open, filtered to Errors:**
1. `wp-admin/` (Dashboard)
2. `wp-admin/post-new.php` (New post)
3. `wp-admin/post.php?post=<ID>&action=edit` (Edit post)
4. `wp-admin/site-editor.php` (Site Editor)
5. `wp-admin/customize.php` (Customizer)
6. `wp-admin/options-general.php` (Settings)

Wait 5 seconds after full load on each page.

**Pass:** Zero `SecurityError` or `DOMException: Blocked a frame` entries.

**Failure indicators:**
- Error trace pointing to `maybe-load-font-family.js`, `get-font-field-css-code.js`, `get-font-family-fallback-stack.js`, or `maybe-load-font-loader-script.js` → Phase 1.5 missed that file
- Error in `dark-mode/index.js` referencing `window.parent` → Phase 1.4 needs a guard

---

### Check 7: Visual spot-check — editor matches frontend

**Templates:** Standard blog post, homepage/index, a page with Nova Blocks content.

**For each:**
1. Open in editor and in a second tab on the frontend
2. Compare:

| Aspect | What to compare |
|--------|----------------|
| Font family | Same typeface in headings and body text |
| Font size/weight | Proportional, matching |
| Colors | Background, text, accent/link colors match |
| Spacing | Margins between blocks consistent |
| Container width | Content column width similar |

**Pass:** Typography, colors identical. No unstyled flash in editor. Block spacing broadly consistent.

**Failure indicators:**
- Editor shows system fonts, frontend shows SM fonts → CSS not injecting into iframe (Phase 1.1)
- Wrong colors in editor → SM custom properties not resolving
- Brief unstyled flash then correct styles → injection timing issue

## Part C: WP 7.0 Beta-Specific Checks

After passing all checks on WP 6.9.1, upgrade to WP 7.0 beta and repeat every check. Additionally:

### C.1 Verify `__unstableResolvedAssets` status

```bash
grep -r "__unstableResolvedAssets" wp-includes/
```
- Found → key still exists (removal not yet happened). Phase 1.1 is still correct but old code would also work.
- Not found → removal confirmed. Phase 1.1 is mandatory.

### C.2 Verify `wp-editor` script handle status

In console on post-new.php:
```js
wp.scripts?.registered?.['wp-editor']
```
- Returns object → still exists (deprecated). Phase 1.2 is future-proofing.
- Returns undefined → removed. Phase 1.2 is mandatory.

### C.3 Check iframe behavior

In post editor DevTools:
1. Check if `<iframe name="editor-canvas">` is present
2. Check if always present (not conditional on theme support)
3. Note any structural changes (different name, nested iframes)

### C.4 Check Font Library API availability

```bash
wp eval "var_dump(function_exists('wp_unregister_font_collection'));"
```
- `true` → Phase 1.7 approach is valid
- `false` or signature changed → Phase 1.7 needs adjustment
